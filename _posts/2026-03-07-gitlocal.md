Every tool that writes configuration to disk faces the same question: what happens when a user commits these files to git? AWS CLI writes credentials to `~/.aws/credentials`, safely outside any repository. But plenty of tools write directly into the project directory, and some of those files contain secrets or machine-specific paths that should never end up in version control.

The tool author knows which files are sensitive. They wrote the code that creates them. Git just doesn't give them any way to say so. The best a tool can do is append a line to `.gitignore`, which means modifying someone else's repository, or hope the user adds the right pattern. When users don't, secrets leak. [git-secrets](https://github.com/awslabs/git-secrets) scans for credential patterns before each commit. [detect-secrets](https://microsoft.github.io/code-with-engineering-playbook/CI-CD/dev-sec-ops/secrets-management/recipes/detect-secrets/) maintains a baseline and alerts on new ones. Blog posts [walk developers through](https://www.twalker.dev/avoid-committing-secrets-by-ignore-tracked-files-in-git/) the various ways to mark tracked files as ignored. These tools exist because the problem is structural: tool authors know a file is sensitive but have no way to express that to git.

Git's existing ignore mechanisms all point in the wrong direction. `.gitignore` requires the repository maintainer to anticipate every tool that might drop files in the project directory. `--assume-unchanged` and `--skip-worktree` are per-clone flags, manual, forgotten after every fresh clone. `export-ignore` in `.gitattributes` only affects `git archive`. I catalogued all of these in my post on [git's magic files](/2026/02/05/git-magic-files.html), and none of them let the file or tool declare itself untrackable. Someone other than the tool always has to act.

A `.gitlocal` marker file would flip this. A tool creates its config directory as usual: `.sometool/config.json`, `.sometool/credentials.json`, whatever it needs. It also drops an empty `.gitlocal` file in that directory. Git sees the marker and ignores the entire directory, no filenames need to change, no comment headers, no format restrictions. It works with JSON, YAML, binary blobs, anything. The pattern follows `.gitignore` and `.gitkeep`, a marker file that changes git's behavior. The name `.gitlocal` isn't used by any existing convention.

The directory marker handles most cases, since tools that write sensitive files usually write them into their own config directory. For standalone files where the tool controls the filename, a `.local.json` or `.gitlocal` extension works: a tool that writes `.sometool/credentials.json` could just as easily write `.sometool/credentials.local.json`. For files where the tool doesn't control the name, a `# gitlocal` comment on the first line covers any format that supports comments: shell, Python, Ruby, YAML, TOML, INI, which is most text config formats where this problem actually shows up. JSON is the main format that can't take a comment, but JSON config files almost always live inside tool-specific directories, so the directory marker covers them.

Credential leaks from git repos happen constantly, and every tool author who writes sensitive files into project directories knows it. Some existing tools might adopt this quickly for exactly that reason. Others, like `.env`, are so established that changing their conventions would take longer. But new tools ship all the time, and their authors currently have to choose between modifying the user's `.gitignore` (presumptuous), documenting which files to ignore (easily missed), or hoping for the best, which is how we got here. `.gitlocal` gives them an option that requires nothing from the user at all.

GitHub has revoked millions of leaked tokens found by scanning public repositories. GitGuardian does the same commercially. These are valuable safety nets, but they catch problems after the fact. The tool author is in the best position to prevent them, because they wrote the code that generates the sensitive file. Right now that knowledge dies at the tool boundary.

The "marker file that changes tool behavior" pattern is well established outside git. The [Cache Directory Tagging Standard](https://bford.info/cachedir/) from 2004 defines a `CACHEDIR.TAG` file that tells backup tools a directory contains regenerable data. GNU tar supports it with `--exclude-caches`. Backup tools like restic, Borg, and rclone all support `--exclude-if-present`, which skips any directory containing a named marker file, exactly the mechanism `.gitlocal` would bring to git. Android's `.nomedia` is probably the most widely deployed example: an empty file in a directory that tells the media scanner to skip it, used by WhatsApp and countless other apps. macOS uses filename suffixes for similar purposes: `.noindex` prevents Spotlight indexing, `.nosync` prevents iCloud sync, `.nobackup` prevents Time Machine backup.

Git already has its own marker files. `.gitignore` tells it what to skip. `.gitkeep` is a community convention for preserving empty directories. `.gitattributes` configures how files are handled during diff, merge, and archive. `.gitlocal` fits naturally into this family as a file that says "this directory is local to this machine."

A file that suppresses its own tracking introduces invisible behavior, and invisible behavior in version control is dangerous, so the semantics have to be tight. `.gitlocal` should only work on untracked directories, never affecting paths that are already tracked. It should show up clearly in `git status` (something like "ignored due to .gitlocal") and be visible through `git check-ignore`. With those constraints it behaves like any other ignore rule, just one that lives closer to the files it describes, the same way `--exclude-if-present` already works in backup tools.

Getting this into git itself means a patch through the git mailing list and convincing the maintainers it belongs in core, a high bar. A more realistic starting point is a `pre-commit` hook that scans for `.gitlocal` markers and prevents staging files in those directories. You'd lose the experience where git just knows, but tool authors could start using the convention immediately and build an evidence base for a core proposal. That's more or less how `.gitkeep` works today: no git code supports it, but the community treats it as standard.

Adding another dotfile to the [growing pile](/2026/02/05/git-magic-files.html) in repository roots is a fair objection. But every leaked secret has a real cost: rotated credentials, audited access logs, sometimes breached systems. An empty marker file in a tool's config directory seems like a reasonable tradeoff. The tools that would benefit most are the ones being written right now, by authors who know exactly which files are sensitive and currently have no good way to act on that knowledge.
