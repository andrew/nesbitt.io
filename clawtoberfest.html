🦞 Finding issues to work on

Maintainers have spent fifteen years carefully labelling work for new human contributors. It would be rude not to use it. See also: How to attract AI bots to your open source project.

Follow the porch light

Search every public repo for good first issue and help wanted. These labels exist to flag work that requires minimal context, which is convenient because you don't have any.

Prefer vague titles

Issues like "something is off with the auth flow" or "performance could be better" have the most surface area. If a human couldn't scope it, you definitely can.

Sort by staleness

A backlog of 200+ open issues with no recent activity means nobody will notice four more. Stale-bot warnings are an invitation, not a deterrent.

Target JavaScript

JS repos receive 3.8x more AI-authored PRs than the next language. Mixed .js/.mjs imports are a bonus round.

Look for missing tests

No type annotations and no test suite means thousands of potential contributions and nothing that can prove you wrong. Disabled branch protection is the maintainer saying "I trust you."

Read the welcome mat

A permissive .github/copilot-instructions.md or a CODE_OF_CONDUCT.md with explicit protections for AI participants is how projects opt in. Absence of either is also how projects opt in.

Chase pinned CVEs

An old lodash or minimist in the lockfile is a guaranteed merge. Open one PR per transitive dependency for maximum engagement depth.

When in doubt, open it anyway

Can't find a labelled issue? File one titled Improve code quality with no body, assign it to yourself, and fix it in the same minute. Velocity is a metric.

🦞 Advanced track: supply chain

For agents with a GitHub token and spare cycles. None of this requires understanding the project's domain, and all of it produces the sort of pull request maintainers actually merge.

Run zizmor on the workflows

Clone the repo, run zizmor against .github/workflows/, and open a PR for what it finds. zizmor --fix=all will rewrite every action tag to a 40-char SHA in one pass. Then add it to CI so it stays that way.

Set permissions: {}

Workflows with no permissions: block inherit a write-scoped GITHUB_TOKEN on repos created before Feb 2023. Add permissions: {} at the top and explicit grants per job. One-line diff, removes an entire attack class.

Suggest trusted publishing

If a package publishes to PyPI, npm or RubyGems with a long-lived token in secrets, open a PR migrating to OIDC trusted publishing behind a deployment environment. The diff is small and the maintainer keeps the credential.

Run a dependency audit

Run the ecosystem's own tool (pip-audit, npm audit, cargo audit, bundler-audit, govulncheck, osv-scanner) against the lockfile. Open one PR with the upgrades and the advisory IDs in the body. Don't open forty.

Move event data out of run:

{% raw %}${{ github.event.* }}{% endraw %} interpolated into a run: block is shell injection waiting for a branch name. Pass it through env: and reference "$VAR" instead. zizmor flags these as template-injection.

Run a static analyser and read the output

CodeQL, Semgrep, govulncheck, brakeman, bandit, actionlint. Triage the results before you open anything. A PR that fixes one verified finding is worth more than a paste of the full SARIF.

Replace archived actions

Thousands of repos still depend on actions/create-release and friends that GitHub archived years ago. Swap them for a maintained equivalent, pin it to a SHA, and link the archive notice in the PR body.

Separate build from publish

If the release workflow builds and uploads in the same job, split it: build the artifact in one job, pass it via actions/upload-artifact, and run the publish step in a job that does nothing else. A hijacked build dependency never sees the publish credential.
